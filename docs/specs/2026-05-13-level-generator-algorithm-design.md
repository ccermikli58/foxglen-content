# Foxglen Level Generator Algorithm — Design Spec

**Date:** 2026-05-13
**Repo:** `foxglen-content/scripts/gen-levels.js`
**Affects:** content-v6 bundle (no binary changes — Unity, RN, Defold all consume content via remote fetch)

---

## Context

`gen-levels.js` ships 5 chapters / 60 levels via a manually-authored `SOURCE` array. Each entry specifies `{ num, arch, diff, kinds, obstacles?, tightness? }` and feeds through `compileLevel(spec)` which deterministically emits `moves`, `goals`, `star1/2/3`, `archetype`, `obstacles`.

**Trigger:** Can reported (2026-05-13) Chapter 1 (L1-9) presents 7 visually identical boards — same `kinds=[0,1,2,3]`, ~no obstacles. Root cause is hand-authored SOURCE that front-loaded onboarding pressure on color variety + obstacle introduction without realising the resulting visual monotony.

**Goal:** Replace hand-authored SOURCE with a deterministic algorithm driven by chapter metadata. Manual overrides remain for "signature" levels (chapter finales, archetype intros, paying-moment walls).

---

## Goals

- New chapters added with **one metadata block + ≤3 override slots**
- Existing L1-60 regenerated by the algorithm (single source of truth, no parallel hand-tuned legacy)
- Algorithm output round-trips through existing `compileLevel()` (no formula/star-math/obstacle-library changes)
- Pacing rules from `app/src/engine/level-design.md` baked into the algorithm (3-tough-then-1-relief, finale tightness, paying-moment placement)
- Algorithm validates: every L's `kinds` ⊆ chapter `spawnKinds`, signature levels respect chapter difficulty band
- Output strictly deterministic: same metadata → identical bundle (no RNG in generator)
- Ch1 visual monotony fix lands as part of v6 by virtue of the algorithm's archetype/obstacle rotation

## Non-goals

- Engine changes (board shape, new obstacles, new specials)
- `compileLevel()` formula changes (goalCount/moves/stars math unchanged)
- Economy node changes
- Atmosphere/friends/UI changes
- Schema bump beyond what the algorithm needs — output stays v5 level shape (`{ num, moves, goals, star1/2/3, archetype, kinds, obstacles }`); bundle envelope bumps from `version: 5` to `version: 6`

---

## Inputs (Chapter metadata extension)

Each entry in `CHAPTERS` gains four new fields:

```js
{
  // existing
  num, start, end, name, biome, spawnKinds, bonus,

  // NEW algorithm-driver fields:
  obstacleTheme: 'none' | 'ice' | 'vine' | 'mix' | 'full',
  difficultyArc: 'gentle' | 'standard' | 'aggressive',
  noviceColor: int | null,    // chapter's "newly introduced" color, gets simpleCollect intro
  overrides: { [levelNum]: Partial<LevelSpec> }   // optional, patches algorithm output
}
```

The hand-authored `SOURCE` array is removed. Algorithm computes the equivalent at build time.

## Outputs

Each chapter produces `length = end - start + 1` LevelSpec entries identical in shape to current SOURCE entries:

```js
{ num, arch, diff, kinds, obstacles?, tightness? }
```

These feed through unchanged `compileLevel(spec)`.

## Algorithm

### Step 1 — Beat allocation

A chapter of length L is partitioned into beats. The pacing template from `level-design.md §7` is parameterised:

```
beats =
  intro:         clamp(round(L * 0.22), 1, 2)
  build_a:       clamp(round(L * 0.30), 2, 4)
  mid:           1
  relief_a:      1
  build_b:       clamp(round(L * 0.25), 2, 4)
  payingMoment:  1   (only if ch.num >= 2)
  relief_b:      1   (only if L >= 13)
  build_c:       remaining - 1
  finale:        1
```

Sum check: if total ≠ L, trim/pad the **latest existing** `build` group (priority: `build_c` if present, else `build_b`, else `build_a`). `intro`/`mid`/`relief`/`payingMoment`/`finale` counts are fixed and never absorb the delta. Always terminates with `finale`.

Beat tags are stable identifiers — every level in the algorithm output carries its beat tag for telemetry + debug.

### Step 2 — Difficulty curve

Per chapter, derive `(diffBase, diffPeak)`:

```
Ch1: base=1, peak=6   (gentle by default)
Ch2: base=3, peak=8
Ch3: base=4, peak=9
Ch4: base=5, peak=10
Ch5: base=6, peak=10
```

`difficultyArc` modifier:
- `gentle`: `peak = max(peak - 1, base + 3)`
- `standard`: as listed
- `aggressive`: `base = base + 1`

Beat → diff mapping inside `[base, peak]`:

```
intro slot i (0-indexed):  base + i
build slot i of N:          lerp(base + 1, peak - 2, i / max(1, N-1))
mid:                        peak - 1
relief:                     max(base, previousLevel.diff - 2)
payingMoment:               peak - 1
finale:                     peak
```

All diff values are integer-rounded.

### Step 3 — Archetype rotation

Per chapter, build an ordered archetype pool from `obstacleTheme`:

```
none:  [dualCollect, tripleCollect, dualCollect, mixed, tripleCollect]
ice:   [iceBreak, mixed, dualCollect, iceBreak, tripleCollect]            // mixed implicitly carries ice via Step 5
vine:  [vineControl, mixed, dualCollect, vineControl, tripleCollect]      // mixed implicitly carries vine via Step 5
mix:   [iceBreak, vineControl, mixed, tripleCollect, scoreOnly]
full:  [pentaCollect, mixed, iceBreak, vineControl, scoreOnly, quadCollect, hexaCollect]
```

**Segment** = a contiguous beat group with the same tag (`build_a`, `build_b`, `build_c` are three separate segments; `intro` is its own segment). Within each segment, archetype rotates by `slotIndexInSegment % pool.length`. Rotation counter resets to 0 at the start of each segment, so `build_a[0]` and `build_c[0]` both pick `pool[0]` — but the diff ramp (Step 2) makes them feel different.

Beat-specific overrides:
- **intro slot 0:** `simpleCollect` if `noviceColor` defined, else `dualCollect`
- **intro slot 1:** `dualCollect`
- **relief:** pick the LAST `simpleCollect`/`dualCollect`/`tripleCollect` from the pool; never `iceBreak`/`vineControl`/`scoreOnly`
- **payingMoment:** pick from pool but force tightness=tight (see Step 6)
- **finale:** chapter-progression archetype:
  - Ch1 → `tripleCollect`
  - Ch2 → `quadCollect`
  - Ch3 → `pentaCollect`
  - Ch4 → `scoreOnly` (peak intensity without object collection)
  - Ch5 → `hexaCollect` (full-palette graduation)
- **build's last slot before finale:** kinds count ramps to `finaleKindsCount - 1` (so a chapter ending pentaCollect has a quadCollect immediately prior)

### Step 4 — Kinds (goal colors)

For each spec, pick goal kinds from chapter's `spawnKinds`:

- **`noviceColor` handling:**
  - If `noviceColor != null` AND slot is `intro` slot 0: `kinds = [noviceColor]`, archetype forced to `simpleCollect`. Highlights the chapter's new color.
  - If `noviceColor == null`: intro slot 0 uses `kinds = [spawnKinds[0]]`, archetype = `simpleCollect`. Intro slot 1 (if it exists) uses `kinds = [spawnKinds[1]]`, archetype = `simpleCollect`. This is the "two-color teaching ladder" for chapter 1 (Mushroom → Clover).
  - For all other slots, `noviceColor` has no effect.
- **Non-intro slots:** walk `spawnKinds` round-robin, preferring kinds **not used in the previous 2 levels** (Hamming-distance optimization on a 2-level lookback window)
- Archetype's `desiredKindCount` (from `compileLevel`: 1/2/3/4/5/6 for simple/dual/.../hexa, 1 for iceBreak/vineControl, variable for `mixed`) determines how many to pick
- For `mixed`: kindsLen = 2 by default, 3 if beat ∈ (mid, payingMoment), 4 if the slot is `build_c`'s last entry (the kinds-ramp-up before finale)
- For `scoreOnly`: emit empty `kinds` (`compileLevel` handles)

Selection is **deterministic on chapter num + slot index** — no RNG.

### Step 5 — Obstacle assignment

Per beat tag (within a chapter, with the chapter's `obstacleTheme` as theme):

```
intro:        no obstacles
build_a:      no obstacles UNTIL the LAST build_a slot — that slot introduces theme's first pattern
mid:          theme's SECOND pattern (medium size)
relief_a:     NO obstacles (psychological slack)
build_b:      theme's third + fourth patterns rotate; for 'mix'/'full' both ice and vine alternate
payingMoment: theme's HARDEST pattern (largest cell count); for 'mix'/'full' both simultaneously
relief_b:     NO obstacles
build_c:      theme's full rotation continues
finale:       signature obstacle stack — 'mix' and 'full' get both ice + vine
```

Pattern library rotation per theme (size-ordered, small → big):

```
ice:   ['ice:corners',  'ice:cross',     'ice:pillars',  'ice:horseshoe',  'ice:border']
vine:  ['vine:scatter', 'vine:lines',    'vine:ring',    'vine:center4']
```

**`mix` theme** alternates ice and vine by slot parity (across all obstacle-bearing slots in the chapter, counter-shared, not per-segment):
- Slot parity 0 → next ice pattern from the ice rotation
- Slot parity 1 → next vine pattern from the vine rotation
- `payingMoment`: forces both `ice:cross + vine:scatter` (medium-size combo)
- `finale`: forces both `ice:horseshoe + vine:ring` (signature stack)

**`full` theme** carries ice + vine simultaneously starting from `mid`. Slot index advances both ice and vine rotation counters in lockstep. `payingMoment` and `finale` use the largest patterns: `ice:border + vine:ring` and `ice:border + vine:center4` respectively.

`vine:center4` (spreader pattern) is reserved for `payingMoment` or `finale` — it ends the level fast if dropped earlier.

### Step 6 — Tightness

Default from existing `defaultTightness(num, arch)` in `gen-levels.js:104`. Beat-tag overrides applied **after** default:

- `payingMoment` → `tight`
- `finale` → `tight` if `peak == 9`, `brutal` if `peak == 10`
- `relief` → `medium` (never `tight`/`brutal`, regardless of default)

If chapter's `difficultyArc == aggressive`, additionally override all `medium` to `tight` from `payingMoment` slot onwards.

### Step 7 — Overrides

After algorithm emits the L-entry spec list for a chapter, iterate the override map and partial-patch:

```js
for (const [num, patch] of Object.entries(chapter.overrides || {})) {
  const idx = specs.findIndex(s => s.num === parseInt(num, 10));
  if (idx < 0) continue;
  specs[idx] = { ...specs[idx], ...patch };   // partial merge, not replacement
}
```

Override use cases (intended):
- Chapter finales that need signature obstacle stacks
- Archetype intros that deserve a specific kind (e.g. L11 first ice signature with Berry as the goal color)
- Paying-moment walls with custom kinds combos
- Curiosity slots (one scoreOnly that lands at a specific story beat)

**Cap:** ≤3 overrides per chapter (warning emitted by validator if exceeded — pure-algorithmic chapters must remain playable).

## Validation rules

Algorithm-emitted specs must satisfy (asserted by new `validateChapter(chapter, specs)` called from `gen-levels.js:run()` before bundle write):

1. Every level's `kinds` ⊆ chapter `spawnKinds`
2. Star thresholds strictly ordered (`compileLevel` already enforces)
3. No 4+ consecutive levels at `diff >= chapter_avg_diff` without a relief level (`diff <= avg - 2`)
4. Chapter finale: `diff == peak`, tightness ∈ {tight, brutal}
5. Relief slots: tightness ∈ {loose, medium}, no obstacles
6. Intro slot 0: archetype ∈ {simpleCollect, dualCollect}, no obstacles
7. First obstacle of a chapter appears in `build_a`'s last slot, not in `intro`
8. Override count ≤ 3 (warning, not error)

Existing `validateBundle` invariants (`gen-levels.js:406-530`) continue to apply on the final compiled bundle.

## Implementation footprint

- **`gen-levels.js` changes:**
  - Add `generateChapter(chapter)` function (~120 LOC)
  - Replace `const SOURCE = [...]` (lines 322-402) with `const SOURCE = CHAPTERS.flatMap(generateChapter);`
  - Add `validateChapter(chapter, specs)` and call after generation, before compile
  - Keep `compileLevel()` unchanged (lines 124-210)
  - `CHAPTERS` array gains `obstacleTheme`, `difficultyArc`, `noviceColor`, `overrides` per chapter
- **`BUNDLE_VERSION`:** 5 → 6, output filename auto-becomes `content-v6.json`
- **Unity + RN content fetch:** Remote Config `content_bundle_url` defaults are hardcoded to v5; either bump default in `ContentBundleRemote.cs:36` / `app/src/content/remote.ts` (binary change) OR set the RC flag explicitly (no binary change). For the v5→v6 rollout we set the RC flag — zero binary touches.

No engine/binary changes required for the core algorithm + content swap. Pure foxglen-content-side commit.

## Risks & mitigations

| Risk | Mitigation |
|------|------------|
| Regenerated levels feel "different enough" to confuse returning players | Bump BUNDLE_VERSION → existing players cold-start into v6. Star history (PlayerProgress) keyed by level num, persists across rebalance. |
| Algorithm produces unwinnable level | Generation-time playability check: for each spec, run `Engine.MakeBoard()` + `HasAnyValidMove` × 5 seeds. If any seed deadlocks, log warning. (Engine has Reshuffle ladder as runtime safety net anyway.) |
| Override drift: people add 5-10 overrides per chapter, defeating the algorithm | Validator warns when overrides > 3/chapter; PR review surfaces it |
| Re-balance shifts star3 ratios unexpectedly | Run preview mode (`gen-levels.js` without `--write`), diff star3 distribution against v5, flag chapters with >20% shift before committing |
| Pacing template wrong for very short (L=5) or very long (L=20+) chapters | Beat allocation clamp+pad logic tested on synthetic L=5/9/12/15/20 in unit tests (`gen-levels.test.js` — new file) |

## Open decisions (resolved inline)

1. **`overrides`: full replacement or partial patch?** → Partial patch (merge semantics). Less footgun: an override of `{ tightness: 'brutal' }` shouldn't have to restate kinds + arch.
2. **`level-design.md` updates?** → Yes. Gains §11 "Algorithmic generation" pointing at this spec + the new chapter-metadata schema. The hand-authoring guidance (§8) stays but is re-framed as "override authoring guidance."
3. **RNG seeding for kinds rotation?** → No. Strictly deterministic on chapter num + slot index. Bundle output must be reproducible for ops + content QA.

## Success criteria

1. Running `node scripts/gen-levels.js` with new CHAPTERS metadata produces 60 levels matching or improving on v5's distribution
2. Ch1 (L1-9) presents at least **4 visually distinct board configurations** (different obstacle patterns + kinds combos visible without examining goal chips)
3. Adding a 6th chapter requires ONLY: append 1 metadata block to `CHAPTERS` + optional `overrides` map. Algorithm produces 15+ playable levels with zero hand-tuning.
4. `validateBundle` passes for both algorithm-only chapters and override-augmented chapters
5. All existing Unity NUnit (253+) + RN engine tests pass against v6 (no engine impact)
6. star3 distribution per chapter within ±20% of v5's per-chapter mean (rebalance is intentional but not violent)

1. Edit `foxglen-content/scripts/gen-levels.js` (add algorithm + CHAPTER fields + validator)
2. `node scripts/gen-levels.js` (preview mode) — compare table with v5, sanity-check pacing
3. `node scripts/gen-levels.js --write` → emit `content-v6.json` alongside the script's parent
4. `git commit && git push origin main` (foxglen-content) → GitHub Pages publishes ~1 min
5. Firebase Remote Config: set `content_bundle_url` to `https://ccermikli58.github.io/foxglen-content/content-v6.json` (currently inherits the hardcoded v5 default)
6. Cold-start clients fetch v6, cache it, activate on next cold start
7. Old binaries with v5 in cache stay on v5 until they cold-start with v6 reachable + RC fetched

NO Unity binary changes. NO RN binary changes. NO Defold binary changes.

## Future extension hooks

- **Per-chapter seed for non-determinism if ever needed** (e.g. weekly event chapter): `chapter.seed: int | null`. Default null = deterministic.
- **Cross-chapter pacing**: a `globalArc` field on the bundle could shift all chapters' diffPeak by ±1 for "easy mode" / "hard mode" remote tuning.
- **Override library**: extract repeated override patterns into named templates (`override:firstIce`, `override:bruteFinale`) — only after we see ≥3 chapters reuse the same patch.
- **Algorithm versioning**: `chapter.algorithmVersion: 1` field. Bumps allow future-different pacing without breaking old chapters reproducibility.
